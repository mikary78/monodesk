## 대시보드 API 설계

### `GET /api/dashboard/summary?year={year}&month={month}`

**응답 구조:**
```json
{
  "year": 2026,
  "month": 3,
  "profit_loss": {
    "total_sales": 12450000,
    "total_expense": 8200000,
    "net_profit": 4250000,
    "profit_margin": 34.1,
    "sales_growth_rate": 8.5,
    "prev_month_sales": 11480000
  },
  "monthly_trend": [ ... ],  // 최근 6개월
  "low_stock_alerts": [ ... ],  // 재고 부족 최대 10건
  "salary_kpi": { ... },
  "recent_expenses": [ ... ],  // 이번 달 최근 5건
  "order_status": { ... }
}
```

## 집계 로직 설명

| KPI | 집계 출처 | 계산 방식 |
|-----|-----------|----------|
| 이번 달 매출 | `sales_records` | cash + card + delivery 합계 |
| 이번 달 지출 | `expense_records` | amount + vat 합계 |
| 순이익 | 계산값 | 매출 - 지출 |
| 전월 대비 증감률 | `sales_records` | (이번달 - 전달) / 전달 × 100 |
| 재고 부족 | `inventory_items` | current_quantity <= min_quantity |
| 급여 현황 | `salary_records` + `employees` | is_paid 집계, resign_date IS NULL = 재직 |
| 발주 현황 | `purchase_orders` | status = '발주중' 건수 |

## 프론트엔드 UI 레이아웃

```
[KPI 카드: 매출] [KPI 카드: 지출] [KPI 카드: 순이익] [KPI 카드: 발주]
[───── 매출 트렌드 차트 (60%) ─────] [── 재고 부족 경고 (40%) ──]
[────── 최근 지출 내역 (50%) ───────] [──── 급여 현황 (50%) ────]
```

## 테스트 및 검증 방법

1. 백엔드 서버 실행 후 `/docs` 에서 `GET /api/dashboard/summary?year=2026&month=3` 호출 확인
2. 각 모듈에 데이터가 없을 때 0/빈 배열로 정상 응답하는지 확인
3. 프론트엔드에서 월 선택 변경 시 데이터 재조회 확인
4. 재고 부족 품목이 없을 때 "재고 부족 품목이 없습니다" 안내 메시지 확인
5. 로딩 중 스켈레톤 UI 정상 표시 확인

## 향후 작업 제안 및 주의사항

- 현재 매출 데이터는 `sales_records` 테이블 기반 (세무/회계 모듈 입력값)
  → POS 연동 시 `sales_analysis` 모듈 데이터도 합산 고려 가능
- 대시보드 API는 단일 엔드포인트로 모든 KPI를 집계하므로,
  데이터가 많아지면 응답 시간이 증가할 수 있음
  → 향후 필요 시 캐싱(Redis 등) 도입 검토
- Step 7 (법인 관리) 완성 후 동업자 지분 현황 KPI 카드 추가 가능
